1. Project Overview

Purpose:
- This frontend is the React client for a To-Do application.
- It currently provides a simple task UI that can list, create, edit, and delete tasks through HTTP calls to the backend.
- The current UI is a single-screen app rendered by App.jsx.

Current functionality:
- Renders a heading: "Todo App".
- Renders a task input field.
- Allows adding a task if the input is not blank after trim().
- Fetches tasks from the backend when the TaskList component mounts.
- Displays tasks returned by the backend.
- Allows entering edit mode for one task at a time.
- Allows saving edited task text if the edit input is not blank after trim().
- Allows canceling edit mode.
- Allows deleting tasks.
- Logs API errors to the browser console.

Backend integration status:
- The frontend is hardcoded to call the backend at http://localhost:3000.
- The task CRUD endpoints are integrated with Axios in src/components/TaskLists.jsx.
- Authentication APIs are not integrated.
- The frontend does not send Authorization headers.
- The current backend task routes are protected by JWT middleware, so the current frontend task API calls will fail with 401 unless the backend is changed or an Authorization: Bearer <token> header is added.
- No token is stored in localStorage, sessionStorage, cookies, React state, or context.
- No register page, login page, protected route handling, or dashboard auth flow exists yet.

Component: TaskList

File:
- src/components/TaskLists.jsx

Purpose:
- Displays task list UI.
- Manages task list state, new task input state, edit mode state, and edit input state.
- Performs all current frontend API calls to the backend /tasks endpoints.

State used:
- tasks
  - Initial value: []
  - Setter: setTasks
  - Purpose: stores task rows returned by GET http://localhost:3000/tasks.
- newTask
  - Initial value: ""
  - Setter: setNewTask
  - Purpose: stores the text typed into the create-task input.
- editingId
  - Initial value: null
  - Setter: setEditingId
  - Purpose: stores the id of the task currently being edited.
  - When editingId equals a task's id, that task renders edit controls instead of normal display controls.
- editTask
  - Initial value: ""
  - Setter: setEditTask
  - Purpose: stores the edited text for the task currently in edit mode.

Props used:
- None.

Effects used:
- useEffect(() => { fetchTasks(); }, [])
- Runs fetchTasks after the component mounts.
- Because main.jsx wraps the app in React StrictMode, this effect may run more than once in development mode depending on React's development behavior.

API calls made:
- GET http://localhost:3000/tasks
  - Function: fetchTasks
  - Response handling: setTasks(response.data)
  - Error handling: console.log(error)
- POST http://localhost:3000/tasks
  - Function: addTask
  - Body: { task: newTask }
  - After success: clears newTask and calls fetchTasks()
  - Error handling: console.log(error)
- DELETE http://localhost:3000/tasks/:id
  - Function: deleteTask
  - Path parameter: id
  - After success: calls fetchTasks()
  - Error handling: console.log(error)
- PUT http://localhost:3000/tasks/:id
  - Function: updateTask
  - Path parameter: id
  - Body: { task: editTask }
  - After success: clears editingId, clears editTask, and calls fetchTasks()
  - Error handling: console.log(error)

Headers sent:
- No custom headers are sent.
- No Authorization header is sent.
- Axios will send JSON content type for POST/PUT request bodies.

Rendered UI:
- Create input:
  - type="text"
  - placeholder="Enter task"
  - value={newTask}
  - onChange updates newTask.
- Add Task button:
  - Calls addTask.
- h2 "Tasks".
- For each task in tasks:
  - The wrapper div uses key={task.id}.
  - If editingId === task.id:
    - Renders edit input controlled by editTask.
    - Save button calls updateTask(task.id).
    - Cancel button clears editingId and editTask.
  - Otherwise:
    - Renders <p>{task.task}</p>.
    - Edit button sets editingId to task.id and pre-fills editTask with task.task.
    - Delete button calls deleteTask(task.id).

Input validation currently implemented:
- addTask returns early if !newTask.trim().
- updateTask returns early if !editTask.trim().

Input validation missing:
- No max length validation.
- No visible validation message.
- No trimming before sending to the backend; the trim check only prevents all-whitespace values.
- No validation for task ids.

Error handling currently implemented:
- API errors are logged to console.

Error handling missing:
- No visible error messages.
- No retry UI.
- No user feedback for 401 Unauthorized.
- No user feedback for failed create/update/delete.

7. State Management

State strategy:
- Local React component state only.
- No Redux, Zustand, MobX, Jotai, Recoil, TanStack Query, SWR, React Context, or reducer pattern is used.
- All state is inside TaskList.

useState usage:
- tasks:
  - Stores the server-returned task array.
  - Updated only by fetchTasks after a successful GET request.
- newTask:
  - Stores the create input value.
  - Updated on every create input change.
  - Cleared after successful POST.
- editingId:
  - Stores which task is currently being edited.
  - Set when user clicks Edit.
  - Cleared after Save or Cancel.
- editTask:
  - Stores the edit input value.
  - Set to task.task when user clicks Edit.
  - Updated on every edit input change.
  - Cleared after Save or Cancel.

useEffect usage:
- TaskList uses one useEffect with empty dependency array.
- It calls fetchTasks when the component mounts.
- No cleanup function is used.
- No AbortController is used to cancel in-flight requests on unmount.
- No dependencies are included because fetchTasks is defined in component scope and the current code intentionally runs the effect only on mount.

Data flow:
1. User interacts with inputs/buttons.
2. Event handlers update local component state or call Axios.
3. Axios sends request to backend.
4. On successful mutation, TaskList calls fetchTasks again.
5. fetchTasks replaces tasks state with the backend response.
6. React re-renders the list from tasks state.

Current refresh strategy:
- Re-fetch after create.
- Re-fetch after update.
- Re-fetch after delete.
- No optimistic updates.
- No cache.
- No polling.
- No stale/revalidate strategy.

Database
