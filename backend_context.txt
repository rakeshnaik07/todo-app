1. Project Overview

Purpose of the application:
- This backend is the API server for a To-Do application.
- It allows users to register, log in, receive a JWT, and manage their own task records.
- Task data is stored in MySQL and is scoped to the authenticated user.

Current functionality:
- Public root health/live route: GET /
- Public user registration: POST /users/register
- Public user login: POST /users/login
- Protected task listing: GET /tasks
- Protected task creation: POST /tasks
- Protected task update: PUT /tasks/:id
- Protected task deletion: DELETE /tasks/:id
- Password hashing with bcrypt before storing user passwords.
- JWT generation during login.
- JWT verification middleware for every task route.
- User-specific task authorization through SQL WHERE clauses using req.user.id.

Main workflow:
1. Client registers a user with username, email, and password.
2. Backend checks whether the email already exists.
3. Backend hashes the password using bcrypt with 10 salt rounds.
4. Backend stores the new user in the users table.
5. Client logs in with email and password.
6. Backend verifies the password with bcrypt.compare.
7. Backend signs a JWT containing the user id and email.
8. Client sends the JWT in the Authorization header as Bearer <token>.
9. authMiddleware verifies the token and attaches the decoded payload to req.user.
10. Task controllers use req.user.id to read, create, update, and delete only that user's tasks.

4. Database Documentation

Database source of truth:
- No migration file, schema file, ORM model, or seed file exists in the backend folder.
- The schema below was verified from the configured MySQL database with a read-only schema query.
- Application code expects two tables: users and tasks.

Current database:
- DB_NAME from .env: todo_app.
- Engine shown by current tables: InnoDB.
- Charset/collation shown by current tables: utf8mb4 / utf8mb4_0900_ai_ci.

Table: users

Columns:
- id
  - Type: int
  - Nullable: no
  - Key: primary key
  - Extra: auto_increment
  - Used as the user identifier in JWT payloads and task ownership checks.
- username
  - Type: varchar(100)
  - Nullable: no
  - Stored from req.body.username during registration.
- email
  - Type: varchar(255)
  - Nullable: no
  - Key: unique
  - Used for registration duplicate checks and login lookup.
- password
  - Type: varchar(255)
  - Nullable: no
  - Stores the bcrypt password hash, not the raw password.

Constraints and indexes:
- PRIMARY KEY (id)
- UNIQUE KEY email (email)
- No explicit created_at or updated_at columns.
- No role, status, verification, reset token, or profile fields.

Current SHOW CREATE TABLE output:

```sql
CREATE TABLE `users` (
  `id` int NOT NULL AUTO_INCREMENT,
  `username` varchar(100) NOT NULL,
  `email` varchar(255) NOT NULL,
  `password` varchar(255) NOT NULL,
  PRIMARY KEY (`id`),
  UNIQUE KEY `email` (`email`)
) ENGINE=InnoDB AUTO_INCREMENT=5 DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_ai_ci
```

Table: tasks

Columns:
- id
  - Type: int
  - Nullable: no
  - Key: primary key
  - Extra: auto_increment
  - Used as the task identifier in update and delete endpoints.
- task
  - Type: varchar(255)
  - Nullable: no
  - Stores the task text from req.body.task.
- user_id
  - Type: int
  - Nullable: yes
  - Stores the owner user id assigned by createTask from req.user.id.
  - Application code treats this as a reference to users.id.

Constraints and indexes:
- PRIMARY KEY (id)
- No foreign key constraint from tasks.user_id to users.id.
- No index currently exists on tasks.user_id.
- user_id is nullable at the database level, even though protected createTask always inserts req.user.id.
- No completed/status, due_date, priority, created_at, or updated_at columns.

Current SHOW CREATE TABLE output:

```sql
CREATE TABLE `tasks` (
  `id` int NOT NULL AUTO_INCREMENT,
  `task` varchar(255) NOT NULL,
  `user_id` int DEFAULT NULL,
  PRIMARY KEY (`id`)
) ENGINE=InnoDB AUTO_INCREMENT=9 DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_ai_ci
```

Relationships:
- Logical relationship: tasks.user_id is intended to reference users.id.
- Database-enforced relationship: none.
- Ownership is enforced in controller SQL, not by a foreign key.

Database operations used by current code:
- SELECT * FROM users WHERE email = ?
- INSERT INTO users (username, email, password) VALUES (?, ?, ?)
- SELECT * FROM tasks WHERE user_id = ?
- INSERT INTO tasks(task, user_id) VALUES (?, ?)
- UPDATE tasks SET task = ? WHERE id = ? AND user_id = ?
- DELETE FROM tasks WHERE id = ? AND user_id = ?

6. Authentication Flow

Registration:
- POST /users/register is public.
- register reads username, email, and password from req.body.
- It queries users by email.
- If the email already exists, it returns 400 with message "Email already exists".
- If the email does not exist, it hashes the password and inserts a user row.
- Successful registration returns 201 with message "User Registered".
- Registration does not automatically log the user in.
- Registration does not return a token.

Password hashing:
- Implemented with bcrypt.hash(password, 10).
- The number 10 is the salt rounds value.
- The stored users.password value is the bcrypt hash.
- The raw password is not intentionally stored by the application.

Login:
- POST /users/login is public.
- login reads email and password from req.body.
- It queries users by email.
- If no row is found, it returns 404 with message "User not found".
- It compares the submitted password with the stored hash using bcrypt.compare.
- If bcrypt.compare returns false, it returns 401 with message "Invalid Credentials".
- If comparison succeeds, it signs a JWT.

JWT generation:
- Implemented with jwt.sign.
- Payload includes:
  - id: user.id
  - email: user.email
- Secret:
  - process.env.JWT_SECRET
- Options:
  - expiresIn: "1d"
- Response includes:
  - message: "Login Successful"
  - token: signed JWT string

JWT verification:
- Implemented in middleware/authMiddleware.js.
- Reads req.headers.authorization or an empty string.
- If missing, returns 401 "No token provided".
- Splits the header by whitespace after trim().
- Expects exactly the scheme Bearer and a token value.
- If scheme is not exactly "Bearer" or token is missing, returns 401 "Malformed authorization header".
- Calls jwt.verify(token, process.env.JWT_SECRET).
- If verification succeeds, decoded payload is assigned to req.user.
- If verification fails for any reason, returns 401 "Invalid Token".

Middleware flow:
1. Request enters Express.
2. cors() runs.
3. express.json() parses JSON bodies.
4. Route matching occurs.
5. Public user routes execute controllers directly.
6. Protected task routes execute authMiddleware first.
7. authMiddleware either returns a 401 response or calls next().
8. Task controller executes with req.user available.
9. Controller performs MySQL query through pool.
10. Controller returns JSON response.

7. Authorization Flow

User-specific tasks:
- Task ownership is represented by tasks.user_id.
- The authenticated user id comes only from the verified JWT payload.
- createTask sets tasks.user_id to req.user.id.
- getAllTasks filters with WHERE user_id = req.user.id.
- updateTask and deleteTask both include AND user_id = req.user.id.

Ownership checks:
- There is no separate ownership lookup.
- Ownership is checked inside the UPDATE and DELETE SQL statements.
- If affectedRows is 0, the API responds with 404 "Task not found".
- This hides whether the task id does not exist or belongs to someone else.

Update restrictions:
- PUT /tasks/:id only updates the task column.
- The SQL requires both id and user_id to match.
- Users cannot update another user's task through the route if the JWT is valid and the secret is secure.
- Users cannot update task id or user_id through this endpoint.

Delete restrictions:
- DELETE /tasks/:id only deletes rows where id and user_id both match.
- Users cannot delete another user's task through the route if the JWT is valid and the secret is secure.
- Non-owned tasks produce the same 404 response as missing tasks.

Authorization limitations:
- The database does not enforce the user/task relationship with a foreign key.
- tasks.user_id is nullable.
- There are no roles or admin permissions.
- There is no route-level authorization beyond "is authenticated".
- The JWT payload is trusted after verification; there is no additional user existence check on each protected request.

9. Middleware Summary

Middleware: cors()
- Registered globally in index.js.
- Current configuration uses cors() with no options.
- This allows broad/default CORS behavior rather than restricting to a known frontend origin.

Middleware: express.json()
- Registered globally in index.js.
- Parses JSON request bodies before route handlers.
- No custom size limit is configured.

Middleware: authMiddleware
- Located at backend/middleware/authMiddleware.js.
- Used only in backend/routes/taskRoutes.js.
- All active /tasks routes are protected by it.
- No /users routes are protected by it.

authMiddleware request lifecycle:
1. Reads Authorization header from req.headers.authorization.
2. If no header exists, returns 401 with "No token provided".
3. Trims and splits the header by whitespace.
4. Expects the first token to be Bearer.
5. Expects the second token to be the JWT.
6. If the format is wrong, returns 401 with "Malformed authorization header".
7. Verifies the token with jwt.verify(token, process.env.JWT_SECRET).
8. If verification succeeds, stores decoded payload on req.user.
9. Calls next().
10. If verification throws, returns 401 with "Invalid Token".

10. Current Application Flow

User Registration
-> Client sends POST /users/register with username, email, password.
-> Backend checks users table for existing email.
-> Backend hashes password.
-> Backend inserts new user.
-> Backend returns "User Registered".

Login
-> Client sends POST /users/login with email and password.
-> Backend selects user by email.
-> Backend compares password with bcrypt hash.
-> Backend signs JWT if credentials are valid.
-> Backend returns "Login Successful" and token.

JWT Token
-> Client stores token.
-> Client sends token in Authorization: Bearer <token>.

Protected Routes
-> Request reaches /tasks route.
-> authMiddleware verifies token.
-> decoded JWT is assigned to req.user.

CRUD Operations
-> GET /tasks reads tasks where user_id equals req.user.id.
-> POST /tasks creates task with user_id equal to req.user.id.
-> PUT /tasks/:id updates task only if id and user_id match.
-> DELETE /tasks/:id deletes task only if id and user_id match.

13. Known Issues

Validation gaps:
- register does not validate required username, email, or password.
- register does not validate email format.
- register does not validate password length or strength.
- login does not validate required email or password.
- createTask and updateTask do not validate that task exists in the request body.
- createTask and updateTask do not reject empty strings.
- createTask and updateTask rely on the database varchar(255) limit rather than checking length before query.
- req.params.id is not validated as a numeric id for update or delete.

Database issues:
- There are no migration files, so schema cannot be recreated from source alone.
- tasks.user_id has no foreign key to users.id.
- tasks.user_id is nullable even though application-created tasks should always have an owner.
- tasks.user_id has no index, which can slow SELECT * FROM tasks WHERE user_id = ? as data grows.
- tasks table has no created_at or updated_at columns.
- users table has no created_at or updated_at columns.

Authentication/security issues:
- Current .env contains secrets and must remain ignored.
- JWT_SECRET existence is not checked at startup.
- Login returns 404 "User not found", which reveals whether an email is registered.
- No rate limiting exists for login or registration.
- No account lockout or brute-force protection exists.
- CORS is globally permissive with default cors() settings.
- JWT has no issuer, audience, or token version claim.
- No token revocation or logout mechanism exists.
- No HTTPS enforcement is handled in the backend.
- No Helmet/security headers middleware is configured.

API behavior issues:
- Message capitalization is inconsistent: "Task created", "Task Updated", "Task Deleted".
- GET /tasks returns user_id to the client.
- No centralized 404 handler exists for unknown routes.
- No centralized error middleware exists for uncaught errors.
- Controllers use SELECT * instead of explicit columns.

Code/architecture issues:
- There is no model or service layer; controllers call SQL directly.
- Repeated try/catch response patterns exist across controllers.
- userController catch blocks do not log errors, while taskController logs errors.
- db.js calls dotenv.config() even though index.js already loads .env.
- DB_PORT exists in .env but is unused.
- PORT is hardcoded instead of using process.env.PORT.
- The commented GET /tasks/:id route references a missing future controller.
